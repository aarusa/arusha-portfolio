A Guide to Monitoring Paused SFMC Automations

  • SSJS Script
  • Data Extension
  • Verification Activity
  • Email Notification

We’ve all been there: a critical data import stops because an automation was "temporarily" paused and never resumed. Instead of manually clicking through folders every Monday morning, I developed a way to log, verify, and alert on paused automations.

The foundation of the framework is a Script Activity using Server-Side JavaScript (SSJS). Instead of checking folders manually, this script programmatically performs a Retrieve on the Automation object. It captures the real-time metadata—specifically the Name, CustomerKey, and Status—of every workflow in your Business Unit and "Upserts" those records into a central staging Data Extension.

This step moves system information into a queryable database. You define a custom Data Extension (e.g., Automation_Status_Audit) to act as your persistent log. By storing numeric status codes (where 2 represents "Paused") alongside a timestamp, you transform abstract platform states into a historical audit trail that can be analyzed using standard SQL.

With your statuses logged into a Data Extension, you use a SQL Query Activity to isolate the outliers. This query filters your master log to identify only the rows where the status is "Paused." The results are then written to a secondary "Alert" Data Extension, effectively creating a real-time "Hit List" of workflows that require manual intervention.

The final step closes the loop by adding a Verification Activity to your monitoring automation. This activity is configured to count the records in your "Alert" Data Extension. If the count is greater than zero, the system automatically triggers an email notification to your team, ensuring that no automation remains paused longer than your defined monitoring interval.
